# Feature Specification: デプロイ方式の変更

**Feature Branch**: `060-change-deploy-method`  
**Created**: 2026-04-04  
**Status**: Draft  
**Input**: GitHub Issue #60 — デプロイ方式の変更

## User Scenarios & Testing _(mandatory)_

### User Story 1 - main ブランチへのマージで自動リリース (Priority: P1)

開発者がリリースブランチ（`release-x.y.z`）またはホットフィックスブランチ（`hotfix-x.y.z`）を main ブランチにマージすると、CI/CD が自動的にブランチ名からバージョンを抽出し、Git タグを作成し、GoReleaser によるリリースを実行する。手動でのタグ作成は不要になる。

**Why this priority**: 現在の手動タグ作成はオペミスのリスクがあり、それを排除することが本 Issue の主目的である。

**Independent Test**: リリースブランチまたはホットフィックスブランチを main にマージした際に、正しいバージョンタグが自動作成され、GitHub Release が公開されることを確認する。

**Acceptance Scenarios**:

1. **Given** `release-1.0.0` ブランチが存在する, **When** main にマージされる, **Then** `v1.0.0` タグが自動作成され、GitHub Releases ページにリリースノートとビルド成果物が公開される
2. **Given** `hotfix-1.0.1` ブランチが存在する, **When** main にマージされる, **Then** `v1.0.1` タグが自動作成され、GitHub Releases ページにリリースノートとビルド成果物が公開される

---

### User Story 2 - 不正なブランチからのマージではリリースしない (Priority: P1)

リリースブランチ以外（feature ブランチなど）が誤って main にマージされた場合、リリースワークフローはリリース処理を実行しない。これにより誤リリースを防止する。

**Why this priority**: オペミス防止が本 Issue の動機であり、誤動作しないことはリリース自動化と同等に重要。

**Independent Test**: feature ブランチを main にマージしてもリリースワークフローがリリースを実行しないことを確認する。

**Acceptance Scenarios**:

1. **Given** `060-change-deploy-method` ブランチが存在する, **When** main にマージされる, **Then** リリースワークフローは失敗して即停止する
2. **Given** `hotfix-urgent` ブランチが存在する, **When** main にマージされる, **Then** リリースワークフローは失敗して即停止する

備考: mainブランチにはミスが重ならないとリリースブランチまたはホットフィックスブランチ以外のブランチからマージされないが、安全装置として、リリースに失敗するようチェックを入れる

---

### User Story 3 - 既存タグとの重複防止 (Priority: P2)

同じバージョンのタグが既に存在する場合、ワークフローは重複タグの作成を試みず、エラーとして処理する。

**Why this priority**: 重複リリースを防止する安全機構として重要だが、正常フローでは発生しにくい。

**Independent Test**: 既に `v1.0.0` タグが存在する状態で `release-1.0.0` をマージし、ワークフローが適切にエラーを報告することを確認する。

**Acceptance Scenarios**:

1. **Given** `v1.0.0` タグが既に存在する, **When** `release-1.0.0` が main にマージされる, **Then** ワークフローはリリースを実行せずエラーを報告する

---

### Edge Cases

- リリースブランチ名がセマンティックバージョニングに従わない場合（例: `release-abc`）はどうなるか？ → バージョン抽出に失敗し、ワークフローはエラーとして終了する
- 同じリリースブランチが再度マージされた場合は？ → 既存タグの重複チェックにより防止される

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: リリースワークフローは main ブランチへの pull request マージ（closed + merged）をトリガーとして起動しなければならない（従来のタグ push トリガーを置き換える）
- **FR-002**: ワークフローはマージされたブランチ名を取得し、`release-` または `hotfix-` プレフィックスで始まるかを判定しなければならない
- **FR-003**: リリースブランチ名からバージョン文字列を抽出し（例: `release-1.0.0` → `1.0.0`）、`v` プレフィックス付きの Git タグ（例: `v1.0.0`）を作成しなければならない
- **FR-004**: 抽出したバージョンがセマンティックバージョニング形式（`X.Y.Z`）に合致しない場合、ワークフローはエラーとして終了しなければならない
- **FR-005**: 同名のタグが既に存在する場合、ワークフローはリリースを実行せずエラーとして終了しなければならない
- **FR-006**: タグ作成後、既存の GoReleaser によるビルド・リリースプロセスを実行しなければならない
- **FR-007**: リリースブランチまたはホットフィックスブランチ以外からのマージでは、リリースワークフローのジョブを実行せず即終了しなければならない

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: リリースブランチの main マージから GitHub Release 公開までが、追加の手動操作なしで完了する
- **SC-002**: リリースブランチ以外からのマージでは、いかなるリリース処理も実行されない
- **SC-003**: 不正なバージョン形式や重複タグの場合、ワークフローが明確なエラーメッセージとともに失敗する

## Assumptions

- リリースブランチ名は `release-X.Y.Z`、ホットフィックスブランチ名は `hotfix-X.Y.Z` の形式に従う（既存のブランチ戦略と整合）
- GoReleaser の設定（`.goreleaser.yaml`）は変更不要。タグが正しく作成されれば既存の設定で動作する
